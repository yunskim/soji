# Soji (소지)

> **소지(素紙):** 아무 장식 없이 본질만 남긴 미색의 캔버스.

Soji는 마크다운 연구 기록과 독립된 데이터 수명주기를 가진 앱을 하나로 묶어 조립하는 파이썬 기반 사이트 관리자입니다. 거창한 SSG 프레임워크 대신, 단 하나의 레지스트리 설정과 스크립트 실행으로 `site/` 트리를 완성합니다.

---

## ⚡ 빠른 시작 (Quick Start)

새로운 콘텐츠를 추가하고 배포하는 과정은 항상 다음 3단계로 끝납니다.

1. **레지스트리 등록**: `registry.yml`에 한 줄 추가
2. **콘텐츠 작성**: `items/` 폴더에 마크다운을 쓰거나 앱 산출물 배치
3. **원클릭 배포**: 터미널에서 `./deploy.sh` 실행

---

## 1. 새 콘텐츠 아이템 추가하기

Soji는 콘텐츠를 노트나 앱으로 나누지 않고, 오직 **데이터가 빌드 시점에 고정되는가**에 따라서만 빌드 방식을 결정합니다.

### 케이스 A: 일반 마크다운 노트 (`build: converter`)
빌드 시점에 텍스트와 고정된 데이터(JSON 등)가 함께 실리는 정적 문서입니다. Tufte-CSS 스타일과 상단 네비게이션이 자동으로 주입됩니다.

1. `items/notebook/` 디렉토리에 마크다운 파일을 생성합니다.
2. 파일 상단에 YAML Frontmatter를 작성합니다. 수식(KaTeX)과 사이드노트를 사용할 수 있습니다.

```markdown
---
title: Hawkes process를 이용한 점 이벤트 모델링
date: 2026-06-14
tags: [statistics, point-process]
meta-description: 점 과정의 기초와 Hawkes process의 금융 데이터 적용 실험
draft: false
---

본문 내용을 여기에 작성합니다. 
인라인 수식 $e^{i\pi} + 1 = 0$ 이나 블록 수식 모두 지원합니다.

맨 위에 붙는 고정 자산(D3 차트용 JSON 등)이 있다면 같은 폴더에 두고 마크다운에서 상대 경로로 호출하면 빌드 시 함께 복사됩니다.

```

### 케이스 B: 자체 데이터 수명주기를 가진 앱 (`build: prebuilt`)

자체 빌드 툴체인(Vite, Webpack 등)을 쓰거나, 콘텐츠 편집과 무관하게 주기적으로 크롤링/샤딩되는 데이터를 다루는 독립적인 아이템입니다.

1. 해당 프로젝트의 자체 툴체인으로 빌드를 완료합니다.
2. 생성된 최종 정적 산출물(HTML, JS, CSS, 대용량 데이터셋) 디렉토리를 그대로 Soji 시스템이 지정한 경로로 복사하거나 해당 경로에서 빌드되도록 설정합니다.
3. Soji는 이 내부 구조를 건드리지 않고 불투명(Opaque)하게 수집하여 지정된 슬러그 위치로 그대로 복사만 수행합니다.

## 2. 레지스트리 설정하기 (`registry.yml`)

모든 콘텐츠의 진실 원본입니다. 새로운 페이지나 섹션을 만들면 반드시 이 파일에 등록해야 배포 파이프라인에 포함됩니다.

```yaml
items:
  # 예시 1: 자체 데이터 수명주기를 가진 앱 (Chrome 주입 안 함)
  - slug: boxoffice
    title: Box Office
    description: 영화진흥위원회 일별 박스오피스 흐름 시각화
    build: prebuilt          # 데이터가 독립적으로 갱신되므로 prebuilt
    chrome: false            # 자체 레이아웃 사용, 헤더 주입 안 함
    nav: true                # 메인 네비게이션에 링크 노출
    order: 10                # 네비게이션 정렬 순서

  # 예시 2: 일반 마크다운 연구 노트 (Tufte 스타일 및 네비게이션 주입)
  - slug: notebook
    title: Notebook
    description: 수학 아이디어·증명·계산 실험 작업 공간
    build: converter         # 마크다운 변환기 통과
    chrome: true             # Tufte-CSS 미색 바탕 및 상단 네비 주입
    nav: true
    order: 60

```

* **`build` 옵션**: `converter` (마크다운 변환) / `prebuilt` (단순 산출물 복사) / 또는 특정 `빌드 명령 문자열`
* **`chrome` 옵션**: `true`로 설정 시 Tufte-CSS 기본 레이아웃과 공통 네비게이션 헤더가 강제 주입됩니다. `false`인 독립 앱은 메인 루트 인덱스 연결만 보장합니다.

## 3. 빌드 및 배포하기

모든 배포는 비공개 저장소에서 빌드되어 결과물만 공개 저장소(`yunskim.github.io`)로 `rsync`되는 구조입니다.

### 로컬 빌드 테스트

배포하기 전 로컬에서 빌드 결과만 확인하고 싶을 때 실행합니다.

```bash
python build.py

```

* `registry.yml`을 읽어 `_drafts/` 폴더나 `draft: true`인 아이템을 제외하고 `site/` 폴더에 최종 결과물 트리를 조립합니다.
* 변환기 아이템들의 날짜를 역순으로 정렬하여 루트 인덱스 페이지를 자동으로 갱신합니다.

### 원클릭 실전 배포

실제 빌드 후 공개 레포지토리 푸시까지 한 번에 완료합니다.

```bash
./deploy.sh

```

**`deploy.sh` 내부 동작 메커니즘:**

1. `python build.py`를 호출하여 최신 변경사항을 `site/`에 조립
2. `site/` 내부 내용을 공개 레포지토리 폴더로 `rsync --delete` 동기화 (이름 변경이나 삭제된 잔재까지 완벽히 클리닝)
3. 공개 레포지토리로 자동 이동 후 `git add -A`, 타임스탬프 커밋 및 `git push` 실행
4. `site/` 내부에 `CNAME`과 `.nojekyll`이 포함되어 있어 깃허브 페이지 인프라가 안전하게 유지됨

---

## 📂 파일 및 폴더 규칙

* **`site/`**: 이 폴더는 배포 시마다 완전히 동기화되므로 손으로 직접 수정하지 마십시오. 오직 `build.py`가 생성하는 결과물만 담깁니다.
* **`draft` 처리**: 발행을 미루고 싶다면 파일 frontmatter에 `draft: true`를 명시하거나 파일명을 `_drafts/` 하위 폴더로 이동하면 빌드 파이프라인이 자동으로 스킵합니다.
