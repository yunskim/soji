# 사이트 콘텐츠 관리 시스템 — 요구사항

`yunskim.github.io`는 Markdown으로 쓰는 연구 기록과, 자체 데이터를 다루는 인터랙티브
시각화를 함께 담는다. 이들을 "노트냐 앱이냐"라는 *정체성*으로 나누지 않는다.
모두 단일 **콘텐츠 아이템**으로 두고, 단 하나의 경계로만 구분한다 —
**아이템에 필요한 데이터가 빌드 산출물에 고정되어 함께 실리는가, 아니면 자체 생산·갱신
수명주기(또는 자체 빌드 툴체인)를 가지는가.** 별도의 정적 사이트 생성기(SSG)는 도입하지
않는다. 작은 **레지스트리 + 빌드 스크립트**가 모든 아이템을 조립해 단일 `site/` 트리로
만든 뒤 공개 저장소로 배포한다. 시스템의 정체성은 "노트 발행기"가 아니라
**사이트 조립·배포 관리자**다.

## 1. 목적·범위

- 모든 콘텐츠 아이템을 **하나의 파이프라인**으로 빌드·배포한다.
- 상단 네비게이션과 루트 인덱스를 **자동 생성**한다(현재는 링크를 매 페이지에 하드코딩, 루트 목록도 수작업).
- 소스·드래프트·빌드 스크립트는 비공개 저장소에, 빌드된 HTML만 공개 저장소에 둔다.
- 가로수길 진행 로그는 이 시스템 위의 한 아이템(컬렉션)으로 들어간다(별도 시스템 아님).
- 비범위: SSG 채택, 백링크·검색·RSS·상태 대시보드 등(9장 참고).

## 2. 콘텐츠 아이템 모델

정체성 이분법(note/app)을 두지 않는다. 모든 공개 대상은 하나의 콘텐츠 아이템이고,
아이템마다 결정할 것은 정체성이 아니라 **빌드 계약**이다.

### 경계: 빌드 시점 고정 자산 vs 자체 데이터 수명주기 (load-bearing)

구분선은 **"런타임에 데이터를 불러오느냐"가 아니다** — 그건 너무 넓다. 정해진 JSON 하나를
브라우저에서 `fetch`해 그리는 페이지도, 그 JSON이 빌드에 함께 실리는 **고정 자산**이라면
본질적으로 정적 문서다. 변환기가 충분히 다룬다.

변환기 바깥(prebuilt)으로 밀어내야 하는 것은 런타임 로딩 자체가 아니라
**자체 데이터 수명주기 또는 자체 빌드 툴체인을 가진** 아이템이다:

- 샤딩된 데이터셋을 커스텀 JS로 스티칭해 불러오거나,
- 콘텐츠 편집과 무관하게 자기 주기로 갱신되는 데이터를 쓰거나,
- 별도 번들러·프레임워크로 빌드돼야 하는 경우.

이런 아이템은 변환기가 소유하려는 순간 데이터 생산·샤딩·갱신 주기까지 떠안아야 하고,
그것이 곧 거부했던 SSG 비대화다.

**판정 테스트 (아이템별 한 줄):**

> 이 아이템에 필요한 모든 것(HTML + 데이터 파일)을 빌드 시점에 고정된 자산으로 함께 실어 보낼 수 있는가?

- 예 → `build: converter` (브라우저에서 그 고정 JSON·CSV를 런타임에 읽어도 무방)
- 아니오 (자체 데이터 수명주기·샤딩 / 자체 빌드 툴체인) → `build: prebuilt` 또는 빌드 명령

### 두 직교 필드

아이템은 정체성이 아니라 다음 두 필드로 기술한다.

- `build:` — 어떻게 빌드되는가. `converter`(MD → Tufte HTML, 고정 사이드카 자산 동봉) / `prebuilt`(미리 빌드된 산출물 수집) / 빌드 명령 문자열.
- `chrome:` — 공유 외형(상단 네비·Tufte 스타일)을 입힐 것인가. `true` / `false`.

전형은 자연히 다음으로 표현된다(이름표는 선택일 뿐이다):

| 콘텐츠 성격 | `build` | `chrome` | 비고 |
| --- | --- | --- | --- |
| 빌드 시점에 고정 (MD 본문, 고정 JSON 읽기 포함) | `converter` | `true` | 공유 네비·스타일 주입. 기존 "노트" + 단순 고정-데이터 시각화 |
| 자체 데이터 수명주기·빌드 (샤딩·독립 갱신) | `prebuilt` 또는 명령 | `false` | 불투명, 자체 데이터 수명주기. 기존 "앱" |

`build`가 `converter`가 될 수 있는지는 임의 선택이 아니라 위 판정 테스트가 결정한다.

## 3. 레지스트리 (콘텐츠 목록)

- 모든 아이템의 **단일 진실원본**인 목록 파일(예: `registry.yml`)을 둔다.
- 필드: `slug`, `title`, `description`, `build`, `chrome`, `nav`(노출 여부), `order`(정렬), `draft`(비공개).
- 정체성 `type` 필드는 두지 않는다. 구분은 `build`/`chrome`가 담당한다.
- 레지스트리는 **콘텐츠 렌더링용 데이터가 아니라 빌드 메타데이터**다. 반복되는 네비·인덱스 수작업을 제거하므로 최소주의 원칙과 충돌하지 않는다.

## 4. 작성 형식 — 변환기 아이템 (`build: converter`)

- 표준 Markdown + YAML frontmatter. 필드: `title`, `date`, `tags`, `meta-description`, `meta-viewport`, `draft`.
- 수식: KaTeX(또는 MathJax). 인라인·블록 모두 지원(프로토타입에서 이미 사용).
- Tufte 핵심 요소 유지: 사이드노트/마진노트, 전폭 그림, 표, 코드블록.
- **고정 사이드카 자산 동봉**: 변환기 아이템은 고정 JSON/CSV/JS를 함께 실어 보낼 수 있어야 하고, `<script>` 임베드를 통과시킨다. 브라우저에서 그 고정 자산을 `fetch`해 D3 등으로 그리는 것은 converter 범위 안이다. (이는 SSG가 아니라 정적 자산 복사다.)
- 데이터가 샤딩되거나 콘텐츠와 무관한 자체 주기로 갱신되기 시작하면 이 경로를 벗어나 `prebuilt`가 된다(2장 경계).
- 아이템 유형(일지/의사결정/실험/데이터 이슈/참고문헌)은 파일을 나누지 않고 (필요 시) 태그로만 구분한다.
- 드래프트 제외: `draft: true`이거나 `_drafts/` 폴더의 파일은 빌드에서 스킵한다.

## 5. 빌드 파이프라인

빌드 스크립트가 레지스트리를 읽어 다음을 수행한다. **SSG 의존성 0.**

1. **변환기 아이템 빌드** (`build: converter`): MD를 Tufte HTML로 변환하고 고정 사이드카 자산을 함께 복사 → `site/<slug>/`. 드래프트 제외.
2. **수집/빌드 아이템 처리** (`build: prebuilt` 또는 명령): `prebuilt`는 산출물 디렉터리를 그대로 `site/<slug>/`로 복사. 빌드 명령이 선언된 경우만 실행 후 복사. 내부 구조·데이터 로딩은 건드리지 않는다.
3. **인덱스 자동 생성**: 변환기 아이템은 frontmatter(`title`,`date`)로 날짜 역순 목록 페이지를 만든다.
4. **조립**: 레지스트리에서 상단 네비 + 루트 인덱스를 생성하고, `chrome: true` 아이템에 주입(6장).
5. 멱등성: 같은 입력은 항상 같은 `site/`를 만든다.

기본 정책상 일상적인 콘텐츠 배포는 자체 수명주기를 가진 아이템 산출물을 재빌드하지 않는다(독립 갱신 주기 보존).

## 6. 사이트 구조·인덱스·네비게이션

- URL 규약: `site/<slug>/...`. 컬렉션 예: `garosu-gil/log/2026-06-04-<slug>/`(최종 위치는 10장).
- **상단 네비게이션**: 레지스트리의 `nav: true` 아이템을 `order`대로 생성. `chrome: true` 아이템에 주입하므로, 항목 추가·삭제가 한 곳에서 끝난다.
- **루트 인덱스**: 현재 손으로 관리하는 프로젝트 목록을 레지스트리에서 생성.
- `chrome: false` 아이템은 불투명 처리한다. 헤더를 강제 주입하지 않고, 자체 내부 네비를 유지한다. 시스템은 루트→아이템 링크와 아이템→홈 복귀 링크 수준만 보장한다.

## 7. 빌드·배포 (private → public, rsync)

구조: **비공개 저장소**가 소스·드래프트·자체 빌드 아이템 소스(또는 산출물)·빌드 스크립트·산출물 폴더 `site/`를 보유한다. `site/`의 내용만 **공개 저장소(`yunskim.github.io`)**로 rsync한 뒤 커밋·푸시한다.

- `site/`를 공개 저장소의 **완전한 진실원본**으로 둔다. `CNAME`, `.nojekyll`까지 `site/` 안에 포함시켜 `--delete` 동기화가 안전하도록 한다.
- 사전 빌드된 HTML이므로 `.nojekyll`로 GitHub의 Jekyll 처리를 끈다(`_` 시작 경로 무시 방지).
- rsync 시 `.git`은 반드시 제외한다(공개 저장소 히스토리 보존).
- 자체 빌드 아이템 산출물도 `site/` 안에 있으므로 동일한 한 번의 rsync로 함께 배포된다.
- 빌드→동기화→커밋·푸시를 단일 `deploy.sh`로 묶어, 공개 저장소는 손으로 건드리지 않는다는 규칙을 강제한다.
- 인지 사항: 공개 저장소 히스토리에는 "재빌드" 기록만 쌓인다. 의미 있는 히스토리는 비공개 저장소에 남는다.
- 자동화 단계(후속): 동일한 흐름을 비공개 저장소의 GitHub Action이 공개 저장소로 푸시(deploy key 또는 PAT)하도록 이전 가능. 현재는 로컬 빌드 후 커밋으로 시작한다.

## 8. 비기능 요구사항

- 모바일 반응형(`meta-viewport` 이미 적용).
- 의존성 최소화. 외부 CDN(KaTeX 등) 사용 정책 명시.
- 합리적 빌드 속도, 빌드 실패·링크 깨짐 검출(최소한 빌드 중단으로 표면화).
- 결합도 최소화: 자체 빌드 아이템은 시스템 없이도 독립 동작 가능해야 한다.

## 9. 추가하지 않는 것 (의도적 제외)

아이템 수가 늘고 "찾기 불편하다"는 실제 불편이 생긴 뒤에 도입한다.

- 백링크/위키링크, 클라이언트 검색(Pagefind 등), RSS/Atom 피드
- 상태 대시보드, 콘텐츠 데이터 파일 렌더링, 콘텐츠 타입별 템플릿 분리, 시리즈 네비게이션
- SSG 전반
- (레지스트리와 변환기의 고정 사이드카 자산 복사는 빌드 메타데이터·정적 자산 처리이므로 이 제외 목록에 해당하지 않는다.)

## 10. 미해결 결정 사항

- **단일 개념의 이름**: 중립적 명칭(`entry`/`item`) vs 가독성을 위한 관용어 유지. (정체성 이분법은 두지 않기로 확정)
- **가로수 컬렉션 위치**: `garosu-gil` 하위 로그 / 기존 `notebook` 활용 / 독립 섹션.
- **콘텐츠 성격**: 빠른 진행 일지 / 정제된 글 / 둘 다 수용.

확정된 것: 경계는 *빌드 시점 고정 자산이냐, 자체 데이터 수명주기·빌드 툴체인이냐*이며,
2장의 판정 테스트로 결정한다. 고정 자산을 런타임에 읽는 것은 `converter` 범위 안이다.

## 부록 A: registry.yml 예시


## 부록 B: deploy.sh 예시

```bash
#!/usr/bin/env bash
set -euo pipefail

SRC="site/"                      # 비공개 저장소의 빌드 산출물 폴더
DEST="../yunskim.github.io/"     # 공개 저장소 (환경에 맞게 조정)

# 1) 빌드: registry.yml을 읽어 converter 아이템 변환(+고정 자산 복사) + prebuilt 아이템 수집 + 인덱스·네비 생성
python build.py

# 2) 동기화: site/ 내용을 공개 저장소로, 삭제분까지 반영, .git 보존
rsync -av --delete --exclude='.git' "$SRC" "$DEST"

# 3) 커밋·푸시
cd "$DEST"
git add -A
git commit -m "site: rebuild $(date +%F)" || echo "변경 없음, 커밋 생략"
git push
```

`SRC`의 후행 슬래시는 "폴더가 아니라 내용물을 복사"를 뜻한다. `--delete`는 이름 변경·삭제된
아이템의 잔재까지 공개 저장소에서 정리한다. `CNAME`·`.nojekyll`이 `site/` 안에 있으므로
이 동기화로도 보존된다.
